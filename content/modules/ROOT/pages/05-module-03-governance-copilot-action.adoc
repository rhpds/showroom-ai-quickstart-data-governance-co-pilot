== Exercise 1: Your first interaction with the copilot

Let's start by exploring the TradeNexus database using natural language queries.

=== Access the copilot interface

The Data Governance Copilot web interface is pre-deployed in your environment.

. Open the Copilot UI in a new browser tab:
+
link:https://copilot-ui-user{guid}-data-gov.{openshift_cluster_ingress_domain}[Copilot UI^]
+
Or navigate directly to:
+
[source,text]
----
https://copilot-ui-user{guid}-data-gov.{openshift_cluster_ingress_domain}
----

. You should see a clean chat interface similar to this:
+
image::screenshot.png[Copilot Interface,align="center",link=self,window=blank,width=700]

=== Explore the database structure

Let's ask the copilot to describe the database.

. In the chat input, type:
+
[source,text]
----
Tell me about this database.
----

. Press Enter or click Send.

. Observe the copilot's response.
+
**What's happening behind the scenes**:
+
* The copilot calls `list_schemas()` to see available schemas
* Calls `list_objects(schema='public')` to enumerate tables
* Calls `get_object_details()` for each table to understand structure
* Synthesizes a natural language summary of the database
+
**Expected response** (may vary based on model):
+
[source,text]
----
This is a PostgreSQL database containing TradeNexus e-commerce data. It includes:

- dim_customer: Customer dimension table with demographic information
- dim_product: Product catalog and details
- fact_sales: Transaction history and sales facts
- Several other dimensional and fact tables

The database follows a dimensional modeling pattern typical for analytics workloads.
----

=== List tables explicitly

Now let's be more specific.

. Ask:
+
[source,text]
----
List the tables and views in the public schema.
----

. The copilot will call `list_objects(schema='public')` and return a structured list.

. Notice how the response includes table names -- this is raw metadata directly from PostgreSQL.

=== Inspect table structure

Let's understand a specific table.

. Query:
+
[source,text]
----
List the details of the dim_customer table in the public schema.
----

. The copilot calls `get_object_details(schema='public', table='dim_customer')` and returns:
+
* Column names and data types
* Constraints (primary keys, foreign keys)
* Indexes
* Any governance metadata in table comments

=== Run an analytics query

Now let's query actual data.

. Ask:
+
[source,text]
----
Visualize the number of customers by state using the dim_customer table.
----

. **Watch the agentic loop in action**:
+
* Copilot inspects `dim_customer` structure
* Identifies the `customer_state` column
* Generates SQL: `SELECT customer_state, COUNT(*) FROM public.dim_customer GROUP BY customer_state`
* Executes the query via `execute_sql()`
* Returns results in a readable format (may include visualization if supported)

. You'll see results showing customer distribution across states.

=== Verify your understanding

**Question**: Why did the copilot inspect the table structure before querying?

**Answer**: The agentic loop validates what's actually in the database rather than guessing. It:

* Confirms the table exists
* Verifies column names (is it `customer_state`, `state`, or `customer_location`?)
* Checks for any constraints or governance annotations
* Ensures the query will succeed before executing

This is fundamentally different from a chatbot that would generate SQL immediately and potentially fail.

== Exercise 2: Governance in action

Now let's see how governance policies are enforced automatically.

=== Inspect governance metadata

The copilot can read governance annotations stored as PostgreSQL comments.

. Query:
+
[source,text]
----
Show me all the metadata associated with the tables and views in the public schema.
----

. The copilot will:
+
* Call `get_object_details()` for each table
* Extract comments that contain governance policies
* Present metadata including:
.. **Certifications**: Is this table "gold standard" or deprecated?
.. **PII fields**: Which columns contain personally identifiable information?
.. **Retention policies**: How long should data be kept?
.. **Last validation date**: When was this table last reviewed?

. **Example metadata you might see**:
+
[source,text]
----
Table: dim_customer
- Certification: gold
- PII fields: customer_email, customer_phone
- Last validated: 2024-03-01
- Retention: 7 years

Table: dim_customer_old
- Status: deprecated
- Replacement: dim_customer
- Retention end: 2024-12-31
----

=== Understanding governance-aware behavior

**How this solves TradeNexus's compliance challenge**:

. **Automated policy discovery**: The copilot reads governance rules from database metadata -- no separate policy management system needed
. **Pre-query validation**: Before executing any query, the copilot can check if the table is certified and safe to use
. **PII awareness**: The copilot knows which fields contain PII and can warn users or filter them automatically
. **Deprecated table avoidance**: The copilot can steer users away from deprecated tables toward current replacements

**What makes this governance-aware instead of just query generation**:

* Traditional chatbot: Generates SQL blindly, might query deprecated tables or PII fields without awareness
* **Data Governance Copilot**: Reads metadata first, validates certifications, respects governance policies, then executes safely

=== Test a governance scenario

Let's simulate a governance check.

. Ask a question that references a table:
+
[source,text]
----
How many customers are in the database?
----

. Observe that the copilot:
+
* Discovers the `dim_customer` table
* Checks if there are multiple customer tables (like `dim_customer_old`)
* Chooses the certified, current table
* Executes the count query

If your environment has deprecated tables marked with metadata, the copilot will avoid them automatically.

== Exercise 3: Understanding the business benefits

Now that you've used the copilot, let's connect the technical implementation to business value for TradeNexus.

=== Benefit 1: Leveraging relational data investments

TradeNexus has years of data in PostgreSQL -- customer history, transactions, product catalogs. Traditional AI solutions often require:

* Exporting data to vector databases for embeddings
* Duplicating data in data lakes
* Moving data to cloud AI services
* Re-architecting storage for AI workloads

**The Data Governance Copilot leverages existing relational data**:

* Queries PostgreSQL directly through MCP
* No data duplication or export
* Preserves existing access controls and governance
* Works with your current schema, no restructuring needed

**Business impact**:

* **Lower costs**: No duplicate storage, no ETL pipelines to AI-specific databases
* **Faster deployment**: Use existing databases immediately
* **Simpler architecture**: Fewer moving parts, less complexity
* **Preserved governance**: Database permissions and policies still apply

=== Benefit 2: Data sovereignty with OpenShift AI

Traditional AI services send your data to external cloud providers:

[source,text]
----
Your data → External API (OpenAI, Anthropic, etc.) → Results return

Risks:
- Data leaves your infrastructure
- Subject to third-party terms of service
- Compliance challenges for regulated industries
- No control over model updates or availability
----

**The Data Governance Copilot achieves data sovereignty**:

[source,text]
----
User query → OpenShift AI (your cluster) → MCP server (your cluster) → PostgreSQL (your cluster)

Everything runs in your controlled environment:
- LLM inference happens on your GPUs
- Data never leaves your cluster
- You control model versions and updates
- Full audit trail in your systems
----

**Business impact for TradeNexus**:

* **Regulatory compliance**: Data stays within controlled boundaries for GDPR, HIPAA, PCI-DSS compliance
* **No vendor lock-in**: You control the AI stack, not dependent on external API providers
* **Predictable costs**: No per-token pricing, no surprise API bills
* **Privacy assurance**: Sensitive customer data never sent to third parties
* **Air-gapped capable**: Can run in disconnected or restricted networks

=== Benefit 3: Self-service analytics without SQL expertise

Before the copilot:

[source,text]
----
Business analyst: "I need to know customer distribution by state"
→ Submit ticket to data team
→ Wait 2-3 days
→ Data analyst writes SQL
→ Results delivered
→ Follow-up questions require repeating the process
----

**With the copilot**:

[source,text]
----
Business analyst: Types "Show me customer distribution by state"
→ Answer in 10 seconds
→ Follow-up questions answered immediately
→ No SQL knowledge required
----

**Business impact**:

* **10x faster insights**: Seconds instead of days
* **Reduced analyst workload**: Data team focuses on strategic work, not routine queries
* **Democratized data access**: Non-technical users can explore data safely
* **Faster decision-making**: Immediate answers enable rapid response to market changes

=== Benefit 4: Built-in compliance and audit trails

Traditional BI tools and SQL access require:

* Manual review of queries for governance violations
* Trust that users won't access PII or deprecated tables
* Separate logging and audit systems
* Compliance checks after the fact

**The copilot provides compliance by design**:

* Every query goes through governance validation
* Automated checks for certifications and PII
* Complete audit trail of tool calls and decisions
* Policy violations prevented, not detected afterward

**Business impact for TradeNexus's compliance audit**:

* **Demonstrable controls**: Show auditors that the system enforces policies automatically
* **Complete audit trail**: Every query logged with governance checks
* **Reduced risk**: Policy violations prevented at execution time
* **Compliance confidence**: Business users can query safely without manual oversight
